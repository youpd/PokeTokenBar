# Companion 캐릭터 상세 설계

작성일: 2026-06-23
대상: TokenMac 성장형 캐릭터 기능
상태: 캐릭터 바이블 초안

## 1. 결론

기본 캐릭터는 **알에서 깨어나는 2D 도트 고양이**로 잡는다.

추천 기본 캐릭터명은 `Mochi`다. 둥글고 부드러운 이름이라 작은 고양이 캐릭터와 잘 맞고, 특정 성별/직업/세계관을 강하게 암시하지 않는다. 앱 안에서는 사용자가 이름을 바꿀 수 있게 하고, 공식 문서와 기본 에셋에서는 `Mochi`를 사용한다.

한 줄 정의:

> Mochi는 AI 코딩 토큰 알에서 깨어나, 사용자의 작업 흔적을 먹고 자라는 작은 메뉴바 고양이다.

이 방향은 새싹/코어 같은 추상형보다 사람을 끌어들이는 힘이 강하다. 고양이는 이미 넓은 호감 기반이 있고, 메뉴바 앱의 상태 표현에도 잘 맞는다. 잠자기, 집중하기, 발로 타이핑하기, 한도에 가까워졌을 때 축 처지기 같은 행동이 직관적이다.

## 2. 왜 고양이가 첫 캐릭터에 맞는가

### 2.1 즉시 이해된다

새싹이나 코어는 설명이 필요하다. 반면 고양이는 18px 메뉴바에서도 귀, 얼굴, 꼬리만 읽히면 바로 캐릭터로 인식된다. 사용자가 처음 앱을 봤을 때 "이게 뭔지"보다 "귀엽다"가 먼저 온다.

### 2.2 업무 도구와도 맞는다

고양이는 조용히 옆에 있는 이미지가 강하다. 계속 말을 거는 캐릭터가 아니라, 책상 위에서 자거나 사용자가 집중할 때 같이 깨어 있는 느낌을 줄 수 있다. 생산성 앱의 분위기를 깨지 않으면서 감정적 존재감을 만든다.

### 2.3 상태 표현이 쉽다

고양이는 작은 프레임 수로도 상태가 잘 보인다.

- Idle: 앉아 있음, 눈 깜빡임
- Working: 앞발로 타이핑
- Focus: 눈이 살짝 좁아짐, 꼬리 움직임
- Tired: 귀가 내려감, 몸이 낮아짐
- Sleep: 몸을 말고 잠
- Level up: 알 조각/별가루/꼬리 흔들림

이 행동들은 설명 없이도 전달된다.

## 3. 알에서 깨어나는 컨셉

포유류가 알에서 깨어난다는 설정은 현실적으로 맞지 않지만, 캐릭터 상품/게임 문법에서는 오히려 기억에 남는다. 중요한 것은 생물학적 일관성이 아니라 "시작점"과 "성장 의식"이다.

### 3.1 Token Egg

Mochi는 일반 알이 아니라 `Token Egg`에서 깨어난다.

Token Egg의 의미:

- 사용자의 첫 AI 코딩 토큰이 응축된 알
- 시작 전에는 아직 어떤 companion인지 모르는 상태
- 첫 사용량이 쌓이면 금이 가고, 귀와 꼬리가 먼저 보임
- 첫 레벨업 때 완전히 깨어남

이 설정은 제품의 핵심 데이터와 캐릭터의 탄생을 연결한다.

### 3.2 알 단계의 장점

알은 온보딩에 좋다.

- 설치 직후 캐릭터가 없어도 허전하지 않다.
- 첫 사용량을 기다리는 이유가 생긴다.
- "곧 깨어난다"는 기대감을 만든다.
- 스크린샷에서 시작/성장 전후가 명확하다.

MVP에서는 설치 직후 `Lv.0 Token Egg`, 첫 XP 지급 후 `Lv.1 Mochi`로 전환한다.

### 3.3 기존 사용량이 많은 사용자

앱을 처음 설치했는데 이미 로컬에 많은 AI 코딩 사용량이 있는 경우에도 첫 화면은 `Token Egg`로 시작한다. 다만 알 상태를 오래 유지하지 않고, 기존 누적 토큰을 읽은 뒤 즉시 부화/성장 연출을 보여준다.

권장 흐름:

1. 첫 실행 시 `Token Egg` 표시
2. `ccusage` 기반 기존 Claude Code / Codex 사용량 스캔
3. "기존 작업 흔적 384.2M tokens를 발견했어요." 같은 짧은 문구 표시
4. 알이 깨지고 `Mochi` 부화
5. 소급 XP를 적용해 현재 누적 사용량에 맞는 레벨까지 빠르게 성장
6. 최종 상태를 `Lv.12 Mochi` 또는 `Lv.30 Terminal Cat`처럼 표시

이 정책은 두 가지 감정을 동시에 만족시킨다.

- 알에서 태어나는 첫 경험은 유지한다.
- 이미 많이 쓴 사용자의 기록을 무시하지 않는다.

처음부터 성장한 고양이를 바로 보여주면 탄생 순간이 사라지고, 반대로 기존 사용량을 무시하고 Lv.0부터 시작하면 앱이 사용자의 기록을 인정하지 않는 느낌이 든다. 따라서 **첫 시각 경험은 알, 최종 상태는 기존 누적 사용량 반영**이 기본 정책이다.

소급 성장 문구 후보:

- "기존 작업 흔적 384.2M tokens를 발견했어요."
- "Mochi가 그동안의 코딩 여정을 기억했어요."
- "Token Egg가 오래 쌓인 토큰에 반응하고 있어요."

소급 XP 적용 후에는 `claimedTokenTotal`을 현재 누적 토큰으로 저장한다. 이후부터는 증가분만 XP로 지급해 같은 과거 사용량이 반복 지급되지 않게 한다.

## 4. 기본 캐릭터: Mochi

### 4.1 형태

Mochi는 32x32 픽셀 캔버스 기준으로 아래 요소를 가진다.

- 머리와 몸: 둥근 고양이 실루엣, 머리/몸을 과하게 분리하지 않음
- 귀: 작은 삼각형 2개, 메뉴바에서 가장 중요한 식별 요소
- 얼굴: 점 눈 2개, 작은 코 1px, 입은 기본 없음
- 수염: 팝오버용에는 1~2px 수염, 메뉴바용에는 생략 가능
- 꼬리: 상태 표현용. 메뉴바에서는 1px 움직임만 허용
- 포인트: 목걸이나 꼬리 끝에 작은 파란색 cursor charm

실루엣 방향:

```text
   /\_/\
  ( o o )
   \___/
    /|
```

실제 32x32 sprite에서는 위 ASCII보다 훨씬 단순하게 만든다. 메뉴바에서는 귀, 둥근 얼굴, 꼬리만 읽히면 충분하다.

### 4.2 성격

Mochi는 사용자를 재촉하지 않는다. 토큰을 더 쓰라고 말하는 캐릭터가 아니라, 사용자가 쌓은 작업 흔적에 반응하는 책상 위 companion이다.

성격 키워드:

- 조용함
- 호기심
- 약간의 엉뚱함
- 방해하지 않는 애정
- 사용자의 작업량을 판단하지 않음

문구 톤:

- "Mochi가 깨어났어요."
- "오늘의 작업 흔적을 조금 먹었어요."
- "지금은 집중 모드예요."
- "한도에 가까워졌어요. Mochi도 잠깐 쉬고 싶어 해요."
- "다음 낮잠 자리까지 2.4M tokens."

피할 문구:

- "더 먹여야 해요"
- "오늘은 부족해요"
- "한도까지 달려요"
- "토큰을 태워요"
- "Mochi가 배고파요"

고양이 캐릭터라도 소비 유도처럼 느껴지는 문구는 피한다.

## 5. 시각 디자인

### 5.1 기본 팔레트

첫 캐릭터는 크림색 또는 연한 회색 고양이가 좋다. 검은 고양이는 매력적이지만 문화권에 따라 상징이 갈릴 수 있고, 주황 고양이는 밈성이 강하다. 크림/회색은 가장 안전하다.

| 용도 | 색상 예시 | 설명 |
|---|---|---|
| 외곽선 | `#2F2F35` | 검정보다 부드러운 먹색 |
| 털 기본 | `#F2E7D5` | 크림색, 라이트/다크 메뉴바 모두 대응 |
| 털 그림자 | `#CDBFA8` | 볼륨감 |
| 털 하이라이트 | `#FFF7EA` | 이마/볼 하이라이트 |
| 귀 안쪽 | `#E9A8A1` | 작은 온기 |
| 눈/코 | `#24242A` | 작은 점 |
| cursor charm | `#7DB7FF` | AI/코딩 연결 포인트 |
| 경고 포인트 | `#F59E0B` | tired 상태에서만 제한 사용 |

색은 귀엽되 과하게 파스텔 장난감처럼 보이지 않게 한다. 외곽선과 그림자를 약간 차분하게 두면 개발자 도구 안에서도 어색하지 않다.

### 5.2 얼굴

얼굴은 강한 캐릭터성을 만들지만 호불호도 만든다. 기본은 낮은 표정 강도가 좋다.

| 상태 | 눈 | 입 | 귀 | 꼬리 |
|---|---|---|---|---|
| Idle | 점 2개 | 없음 | 기본 | 천천히 1px |
| Working | 점 2개 | 없음 | 기본 | 좌우 1px |
| Focus | 눈 1px 좁아짐 | 없음 | 살짝 앞으로 | 거의 고정 |
| Tired | 눈 아래로 1px | 짧은 선 가능 | 내려감 | 낮게 |
| Sleep | `- -` | 없음 | 낮게 | 몸 옆 |
| Level up | 눈 2px 확장 | 작은 선 가능 | 올라감 | 빠르게 1회 |

큰 눈, 큰 입, 과장된 울상은 피한다. `귀`와 `꼬리`만으로도 충분히 감정이 보인다.

### 5.3 메뉴바 식별성

메뉴바 18px에서 고양이로 보이려면 아래 3개가 우선이다.

1. 귀 2개
2. 밝은 둥근 얼굴
3. 외곽선

수염, 목걸이, 무늬는 메뉴바에서는 생략해도 된다. 팝오버 sprite에서만 디테일을 살린다.

## 6. 성장 단계

Mochi는 크기가 커지는 것보다, 알에서 깨어나고 자세와 디테일이 늘어나는 방식으로 성장한다. 메뉴바의 정체성을 유지하려면 귀와 둥근 얼굴 실루엣은 계속 유지한다.

### 6.1 Lv. 0: Token Egg

역할: 설치 직후 / 사용량 대기 상태.

특징:

- 크림색 알
- 파란색 작은 token dot
- 금 없음 또는 아주 작은 빛
- 메뉴바에서는 알 아이콘처럼 보임

느낌:

> 아직 깨어나지 않은 companion.

### 6.2 Lv. 1: Hatchling Mochi

역할: 첫 탄생.

특징:

- 알이 깨지고 작은 귀가 보임
- 얼굴은 점 눈 2개만
- 꼬리는 아직 짧음
- idle blink만 있음

느낌:

> 토큰 알에서 나온 첫 companion.

### 6.3 Lv. 5: Desk Kitten

역할: 첫 성장 체감.

특징:

- 알 조각을 작은 방석처럼 깔고 앉음
- 꼬리가 명확해짐
- working 상태에서 앞발 2프레임 움직임

느낌:

> 이제 메뉴바 책상에 자리 잡은 고양이.

### 6.4 Lv. 15: Cursor Cat

역할: AI 코딩 companion 정체성 강화.

특징:

- 작은 파란 cursor charm이 목에 생김
- focus 상태에서 cursor charm이 깜빡임
- 앞발 타이핑 애니메이션이 4프레임으로 늘어남

느낌:

> 사용자가 코딩할 때 같이 깨어 있는 고양이.

### 6.5 Lv. 30: Terminal Cat

역할: 장기 사용 보상.

특징:

- 알 조각 방석이 작은 터미널 매트처럼 변함
- 꼬리 끝에 작은 파란 포인트
- tired 상태에서 매트 위에 엎드림

느낌:

> 사용자의 작업 리듬을 아는 companion.

### 6.6 Lv. 50: Orbit Cat

역할: 기본 시즌 완성형.

특징:

- 주변에 작은 token dot 1~2개가 천천히 돎
- 메뉴바에서는 token dot을 생략하거나 1px 포인트만 사용
- 팝오버 level up에서 알 조각과 token dot이 반짝임

느낌:

> 오래 쌓인 작업 흔적이 고양이 주변의 작은 궤도로 남음.

## 7. 상태별 애니메이션

### 7.1 메뉴바 프레임 수

| 상태 | 프레임 | 주기 | 설명 |
|---|---:|---:|---|
| Egg | 2 | 3~5초 | token dot 약한 점멸 |
| Idle | 2 | 3~5초 | 눈 깜빡임 |
| Working | 4 | 0.8~1.2초 | 앞발 타이핑 |
| Focus | 4 | 0.5~0.8초 | cursor charm 점멸 |
| Tired | 2 | 1.5~2초 | 귀 내려감 |
| Sleep | 2 | 2~4초 | 몸 말고 잠 |
| Level up | 6 | 1회 | 알 조각/별가루 반짝임 |

### 7.2 상태 결정

기존 `UsageStore.spinTier`와 한도 정보를 그대로 활용한다.

| 캐릭터 상태 | 조건 |
|---|---|
| Egg | companion 미부화 또는 XP 0 |
| Sleep | 오늘 사용량 0 또는 장시간 변화 없음 |
| Idle | burn rate 낮음 |
| Working | burn rate 보통 |
| Focus | burn rate 높음 |
| Tired | Claude/Codex 한도 임계값 근접 또는 소진 예측 |
| Level up | XP 지급 후 level 증가 |

`Tired`는 사용자를 혼내는 상태가 아니라 "쉬어도 된다"는 신호다. 빨강보다 amber를 우선 사용하고, 실제 한도 경고 UI만 빨강을 쓴다.

## 8. 메뉴바 사양

> 갱신(2026-06-23): 아트 방향이 소프트 벡터 **코드 드로잉**으로 확정되어(§15.4), 아래 PNG 기반 항목은 코드 드로잉으로 대체한다.

권장 렌더링(코드 드로잉):

- 렌더: Core Graphics / SwiftUI Path로 직접 그림(벡터, 안티에일리어싱). PNG sprite 아님.
- 표시 크기: 18~24px(메뉴바 높이에 맞춤). 벡터라 retina에서 선명.
- 텍스트와 함께 표시 시 아이콘 폭 고정.
- 움직임: 팔 없이 귀·꼬리·몸통·표정. 메뉴바에서는 진폭을 작게.
- 메뉴바/팝오버는 같은 드로잉 함수를 크기만 다르게 호출(스프라이트 분리 불필요).

메뉴바에서 유지할 요소:

- 귀 2개
- 둥근 얼굴
- 외곽선
- 상태별 1px 움직임

메뉴바에서 생략할 요소:

- 수염
- 자세한 발
- 알 조각의 세부 균열
- token dot 여러 개
- 작은 목걸이 디테일

## 9. 팝오버 사양

팝오버에서는 캐릭터가 감정적 중심이 된다.

추천 레이아웃:

```text
┌────────────────────────────────┐
│ [ Mochi sprite ]  Mochi Lv. 12 │
│                  ███████░░░    │
│                  오늘 +148 XP  │
│                  다음 낮잠 자리 │
│                  2.4M tokens   │
└────────────────────────────────┘
```

상단 캐릭터 영역은 과한 카드가 아니라 현재 팝오버의 첫 섹션으로 자연스럽게 둔다.

정보 우선순위:

1. 캐릭터
2. 이름과 레벨
3. XP progress
4. 오늘 획득 XP
5. 다음 해금까지 남은 토큰
6. 짧은 상태 문구

기존 토큰 상세와 한도 상세는 아래에 유지한다.

## 10. 해금과 스킨

### 10.1 기본 해금

| 조건 | 보상 |
|---|---|
| 첫 XP 지급 | Token Egg 부화 |
| Lv. 5 | Desk Kitten |
| Lv. 15 | cursor charm |
| Lv. 30 | terminal mat |
| Lv. 50 | orbit token dot |
| 7일 streak | 작은 별 모양 목걸이 |
| 누적 100M tokens | night palette |
| 누적 1B tokens | prism cursor charm |

### 10.2 스킨 원칙

스킨은 고양이의 정체성을 바꾸지 않고 색과 작은 포인트만 바꾼다.

- cream: 기본, 가장 호불호 적음
- gray: 차분한 업무 도구 감성
- calico: 귀엽지만 기본값으로는 두지 않음
- black: 매력적이지만 문화권 상징이 갈릴 수 있어 unlock skin
- night: dark teal + muted blue
- mono: grayscale

강한 취향의 스킨은 기본값으로 두지 않는다.

## 11. Max 성장과 새 알

Mochi가 max level에 도달하면 사라지지 않는다. 고양이가 사라지면 사용자는 지난 성과를 잃었다고 느낄 수 있다. 따라서 max 달성은 종료가 아니라 `Collection`으로 옮겨지는 졸업에 가깝게 표현한다.

권장 흐름:

1. `Mochi`가 max level 달성
2. `Orbit Cat` 완료 연출 표시
3. `Mochi`에 `Maxed` 배지 부여
4. `Mochi`를 Collection에 보존
5. 새 `Token Egg` 지급
6. 다음 XP는 새 알에 적용
7. `Mochi`는 Playground에서 계속 뛰놀거나 대표 캐릭터로 다시 지정 가능

완료 연출:

- token dot이 고양이 주변을 한 번 돌고 사라짐
- 알 조각 방석이 작은 배지로 변함
- 문구: "Mochi가 max level에 도달했어요."
- 문구: "새 Token Egg가 도착했어요."

새 알은 아직 키워보지 않은 다른 캐릭터를 우선 부화시킨다. 사용자가 새 알을 얻었는데 같은 캐릭터가 바로 나오면 보상감이 약해진다. 모든 기본 캐릭터를 모은 뒤에만 variant, rare skin, seasonal egg를 허용한다.

## 12. Collection / Playground

메뉴바 클릭 팝오버에는 캐릭터를 보관하고 다시 볼 수 있는 탭이 필요하다.

### 12.1 Collection

Collection은 도감이자 성과 보관함이다.

표시 항목:

- 캐릭터 sprite
- 이름
- 레벨 / max 여부
- 부화 날짜
- max 달성 날짜
- 누적 XP
- 선택한 skin
- 대표 캐릭터 지정 버튼

Collection에서 중요한 것은 "내가 키운 캐릭터가 사라지지 않았다"는 확신이다. maxed 캐릭터는 흐리게 처리하거나 비활성화하지 않는다. 오히려 작은 crown, star, token badge처럼 완료 표시를 준다.

### 12.2 Playground

Playground는 보유 캐릭터들이 자유롭게 움직이는 작은 공간이다. [git-goods/gitanimals](https://github.com/git-goods/gitanimals)의 farm mode처럼 여러 캐릭터가 한 화면에 존재하는 감정적 보상을 로컬 앱 팝오버 안으로 가져온다.

MVP 이후 권장 표현:

- 작은 책상/터미널 배경
- 보유 캐릭터 3~8마리가 천천히 걷거나 잠
- active 캐릭터는 살짝 더 선명하게 표시
- maxed 캐릭터는 badge를 달고 자유롭게 돌아다님
- 클릭하면 해당 캐릭터 상세로 이동

기술적으로는 정교한 물리나 AI가 필요 없다. 각 캐릭터에 idle/walk/sleep 상태와 좌우 방향만 주고, 몇 초마다 목표 위치를 바꾸면 충분하다. 핵심은 게임성보다 "내 캐릭터들이 모여 있다"는 감정이다.

## 13. 두 번째 캐릭터 세트

두 번째 캐릭터도 고양이 계열로 가는 편이 낫다. 첫 릴리스에서 종을 넓히면 정체성이 흐려진다. 고양이 안에서 실루엣과 성격을 바꾸는 것이 더 안정적이다.

추천 두 번째 세트:

`Byte`: 회색 short-hair 계열 고양이.

특징:

- Mochi보다 조금 각진 얼굴
- 더 차분한 회색 팔레트
- cursor charm 대신 작은 terminal cursor tail
- 성장하면 꼬리 끝에 파란 prompt 점이 생김

역할:

- 크림색 귀여움이 취향이 아닌 사용자를 위한 차분한 대안
- 개발자 도구 감성 강화
- 같은 고양이 세계관 유지

해금 조건:

- Mochi max 달성 후 지급되는 새 Token Egg
- 또는 누적 50M tokens를 넘긴 첫 설치 사용자의 초기 소급 성장 보상

## 14. 캐릭터 문구 가이드

Mochi는 사용자에게 명령하지 않는다. 소비를 유도하는 말보다 상태를 알려주는 말이 좋다.

### 14.1 좋은 문구

- "Mochi가 깨어났어요."
- "오늘의 작업 흔적이 쌓였어요."
- "Mochi가 한 단계 자랐어요."
- "다음 낮잠 자리까지 2.4M tokens."
- "지금은 집중 모드예요."
- "한도에 가까워졌어요. 잠깐 쉬어도 괜찮아요."
- "오늘은 조용히 자리를 지키고 있어요."
- "Mochi는 Collection에서 계속 만날 수 있어요."
- "새 Token Egg가 도착했어요."

### 14.2 피할 문구

- "더 달려요."
- "토큰을 더 먹여요."
- "Mochi가 배고파요."
- "오늘은 부족해요."
- "한도까지 얼마 안 남았어요, 계속 가요."
- "게으름 상태."

문구의 기준은 "소비 유도"가 아니라 "기록과 상태 안내"다.

## 15. 구현용 에셋 명세

### 15.1 파일 구조

런타임 렌더는 **PNG atlas + manifest**다. 개별 프레임 PNG 수백 장 대신, 캐릭터별 atlas 1장(상태/단계 행) + 프레임 사각형 manifest로 둔다. `AppDelegate.advanceSpin`이 이미 캐시된 NSImage를 타이머로 교체하는 구조라(코인과 동일 패턴), atlas에서 잘라낸 NSImage를 같은 흐름에 얹는다. **메뉴바(18px)와 팝오버(96~128px) atlas는 분리**한다(§8/§9) — 18px는 다운스케일이 아니라 별도 큐레이션 세트.

```text
assets/companions/
  mochi/
    metadata.json
    menubar-atlas.png       # 18px 프레임 행 (소량, 손튜닝)
    menubar-manifest.json   # 프레임 사각형 + 애니 메타
    popover-atlas.png       # 96~128px 프레임 행 (AI 생성)
    popover-manifest.json
  byte/
    metadata.json
    ...
```

atlas + manifest 포맷은 생성기를 바꿔도 유지되는 표준 형태라 저후회 선택이다(15.4).

### 15.2 metadata 예시

```json
{
  "id": "mochi",
  "displayName": "Mochi",
  "defaultUnlocked": true,
  "species": "token_cat",
  "stages": [
    { "minLevel": 0, "name": "Token Egg" },
    { "minLevel": 1, "name": "Hatchling" },
    { "minLevel": 5, "name": "Desk Kitten" },
    { "minLevel": 15, "name": "Cursor Cat" },
    { "minLevel": 30, "name": "Terminal Cat" },
    { "minLevel": 50, "name": "Orbit Cat" }
  ],
  "states": ["egg", "idle", "working", "focus", "tired", "sleep", "levelUp"]
}
```

### 15.3 필요한 첫 에셋 수

MVP에서 모든 레벨/상태 조합을 만들 필요는 없다.

최소 세트:

- `lv0_egg`: 2 frames
- `lv1_idle`: 2 frames
- `lv5_idle`: 2 frames
- `lv5_working`: 4 frames
- `lv15_focus`: 4 frames
- `lv15_tired`: 2 frames
- `lv15_sleep`: 2 frames
- `level_up`: 6 frames

총 24장으로 MVP를 시작할 수 있다. 팝오버용은 같은 원본을 확대해도 되지만, 가능하면 대표 이미지 6장을 별도로 둔다.

### 15.4 에셋 제작 방식 (확정: 코드 드로잉 소프트 벡터, 2026-06-23)

**아트 방향 확정: Pusheen 계열 소프트 벡터.** 픽셀아트(도트)가 아니라 부드러운 둥근 벡터다. 여러 시안 반복 끝에 사용자가 "Pusheen 같은 느낌"으로 확정.

근거(폭넓은 레퍼런스 조사):

- **베이비 스키마**(Lorenz, Kindchenschema): 큰 머리·둥근 몸 + 크고 낮고 멀리 떨어진 반짝이는 눈 + 작은 코·입 + 둥근 볼이 귀여움의 핵심 트리거. ([PMC](https://pmc.ncbi.nlm.nih.gov/articles/PMC3260535/), [의인화 비율 연구](https://www.sciencedirect.com/science/article/pii/S1875952123000411))
- **Pusheen·Hello Kitty 교훈**: 화려함보다 단순·둥글기·공감이 사랑받는다. 큰 눈도 부드러운 색·깔끔한 형태로 두어 부담을 줄인다. ([CuteStudies](https://www.cutestudies.com/compendium/pusheen))
- **귀 규칙**: 귀를 따로 붙이지 않고 머리+귀+몸을 하나의 실루엣으로(앞면이 머리 윤곽을 파고듦), 머리 위 1/3에서 모서리로 벌림, 밑 넓고 위로 좁은 삼각, 변은 볼록, 끝은 살짝 둥글게. ([easydrawingguides](https://easydrawingguides.com/how-to-draw-a-chibi-cat/), [MediBang](https://medibangpaint.com/en/use/2023/05/cute-animal-ears/))

**렌더링: 코드 드로잉(Core Graphics / SwiftUI Path), PNG 스프라이트 아님.**

- 캐릭터를 상태 파라미터(breathe/bounce/earUp/droop/eye/sway/charm/loaf …)로 코드에서 직접 그린다. 팔(앞발) 없음 — 움직임은 귀·꼬리·몸통·표정으로만.
- **작동하는 레퍼런스 구현이 이미 있다**: `patrick-html/2026-06-23-mochi-state-preview.html`의 JS(`drawMochi`/`drawCat`/`earPath` …). 이 로직을 Swift(Core Graphics/SwiftUI Path)로 포팅한다.
- 메뉴바/팝오버는 같은 드로잉을 크기만 다르게 렌더(벡터라 해상도 무관, 안티에일리어싱). 메뉴바는 retina에서 선명.

**이 결정이 대체하는 것:**

- 이전의 sprite-gen + atlas/manifest + 픽셀 양자화 파이프라인(§15.1~15.3, [companion-art-prompts.md](companion-art-prompts.md))은 **불필요**. 단순 소프트 벡터는 코드 생성이 더 적합하고, 1인 OSS 아트 병목(risk 10.3)도 해소된다.
- Phase 1의 "PNG 번들 vs 코드 드로잉" 미결정은 **코드 드로잉으로 확정**. SwiftPM `resources:` PNG 번들 불필요.
- §8 메뉴바 사양의 "32x32 PNG / interpolation .none"은 코드 드로잉으로 갱신(아래 §8 주석 참조).

진화 단계·시즌 캐릭터·스킨도 같은 드로잉 함수에 파라미터(팔레트/마킹/액세서리)를 더하는 방식으로 확장한다(별도 에셋 생성 없음).

**종·품종 확장은 [companion-design-system.md](companion-design-system.md)로 분리.** 하나의 공통 엔진(만두 베이스 + 베이비 스키마)에 트레이트(귀 pointy/floppy/fluffy · 주둥이 · 꼬리 curl/wag/plume · 마킹 tabby/points/saddle/curlyfluff · 팔레트 · 비율)만 바꿔 고양이(크림 태비/그레이/샴)·강아지(리트리버/푸들/닥스훈트) 등으로 확장한다. 새 품종은 트레이트 표 한 줄로 추가.

## 16. 아트 제작 프롬프트

> 전체 키트(base · 상태별 스트립 · 메뉴바 18px 변형 · sprite-gen 액션 리스트 · 후처리)는 [companion-art-prompts.md](companion-art-prompts.md). 아래는 단일 base 예시.

초기 AI 이미지 생성이나 외주 요청에 쓸 수 있는 프롬프트:

```text
Create a tiny pixel-art cat companion for a macOS menu bar app.
The character is a cream-colored kitten named Mochi that hatches from a glowing token egg.
It should feel calm, cute, and suitable for a developer productivity tool.
32x32 pixel canvas, limited 8-color palette, soft dark outline, cream fur,
small triangle ears, tiny dot eyes, tiny nose, no large mouth, subtle blue cursor charm.
Include a cracked token egg stage and a small hatched kitten stage.
Design must remain readable when scaled down to 18x18 pixels.
Avoid anime-style large eyes, aggressive expressions, complex accessories, and overly childish baby proportions.
```

한국어 버전:

```text
macOS 메뉴바 앱에 들어갈 작은 픽셀 아트 고양이 companion 캐릭터를 만든다.
캐릭터 이름은 Mochi이고, 빛나는 token egg에서 깨어나는 크림색 아기 고양이다.
개발자 생산성 도구에 어울리도록 차분하고 귀엽게 만든다.
32x32 픽셀 캔버스, 8색 이내 팔레트, 부드러운 어두운 외곽선, 크림색 털,
작은 삼각형 귀, 작은 점 눈, 작은 코, 큰 입 없음, 작은 파란색 cursor charm.
금이 간 token egg 단계와 깨어난 kitten 단계를 포함한다.
18x18 픽셀로 줄여도 귀와 얼굴 실루엣이 읽혀야 한다.
애니풍 큰 눈, 공격적 표정, 복잡한 액세서리, 지나친 아기 비율은 피한다.
```

## 17. 시즌 한정 캐릭터

특정 시기에만 부화하는 히든 캐릭터다. 일상 로스터(Mochi/Byte)와 달리, 그 시즌에 앱을 켜 둔 사용자만 만난다. 컬렉션 도감(§12)에 "12월 한정" 같은 항목이 남아 수집·공유 동기를 만든다.

### 17.1 정체성 원칙

§13의 "고양이 안에서 실루엣/액세서리만 바꾼다"를 그대로 이어간다. 시즌 캐릭터는 새 종이 아니라 **고양이 + 시즌 액세서리 + 시즌 팔레트**다. 메뉴바 18px에서 귀/둥근 얼굴/외곽선 식별성(§2.1, §5.3)을 유지한 채 시즌감을 얹는다.

- "눈사람"이 아니라 **Snow Cat**: 눈사람 모자 + 머플러 + 코끝 1px 당근색.
- "루돌프"가 아니라 **Rudolph Cat**: 작은 뿔 + 빨간 코 1px + 갈색 팔레트.

### 17.2 시즌 로스터

로컬 시스템 날짜 기준 윈도(1인 로컬 단일 기기 전제와 일치). 윈도 밖에서는 풀이 비활성이라 지급되지 않는다.

| 시즌 | 윈도(로컬) | 캐릭터 | 등급 | 모티프 |
|---|---|---|---|---|
| 겨울 홀리데이 | 12/1–12/31 | Rudolph Cat | rare | 뿔, 빨간 코, 갈색 |
| 겨울 홀리데이 | 12/1–12/31 | Snow Cat | unique | 눈사람 모자, 머플러, 흰/하늘 |
| 새해 | 1/1–1/15 | Fortune Cat | rare | 복(福) 부적, 금색 포인트 |
| 봄 | 3/20–4/30 | Sakura Cat | rare | 벚꽃잎, 연분홍 |
| 여름 | 7/1–8/20 | Beach Cat | rare | 수박/튜브, 바다색 |
| 가을·할로윈 | 10/1–10/31 | Pumpkin Cat | unique | 호박 모자, 주황/단풍 |
| 겨울(희귀 정점) | 12/20–12/31 | Aurora Cat | legendary | 오로라 글로우, 미세 token dot |

`Aurora Cat`은 시즌 legendary 예시다. 겨울 윈도 안에서도 더 좁은 구간 + legendary 게이트(§18)가 겹쳐 가장 늦게 나온다.

### 17.3 지급 정책

- **토큰으로 살 수 없다.** 알은 진화/streak/누적 마일스톤(§10.1)으로만 지급된다. 시즌 윈도 중 첫 알 지급 1회를 시즌 알로 확정한다. 기준은 소비량이 아니라 "그 시기에 켜 두었나(presence)"다. 이는 과소비 장려 리스크(ideation §10.1)를 정면으로 막는다.
- **시즌당 1회.** `seenSeasons`에 `winter_2026`처럼 기록해 같은 시즌 내 중복 지급을 막는다.
- **매년 재등장.** 시즌 캐릭터는 매년 같은 시기에 다시 받을 수 있다. 한 해를 놓쳐도 영구 상실이 아니다. 비수익 1인 OSS 가치에 맞춰 FOMO 압박을 최소화한다.
- 이미 보유한 시즌 캐릭터가 다음 해에 다시 뽑히면 `알 조각(shard)`으로 환산한다(§18).

문구 후보:

- "12월에만 깨어나는 Token Egg가 도착했어요."
- "Rudolph Cat이 겨울 책상에 찾아왔어요."
- "이 시즌 캐릭터는 내년에 다시 만날 수 있어요."

## 18. 등급과 확률

알에서 어떤 캐릭터가 나오는지는 **등급(rarity) 가중치**로 조절한다. 흔한 캐릭터는 빨리, 어려운 캐릭터는 늦게 나오게 한다. 상세 컨셉은 ideation §5.6과 동일하며, 여기서는 캐릭터 바이블 관점의 등급 배치만 정리한다.

### 18.1 등급 사다리

| 등급 | 배치 | 예시 |
|---|---|---|
| normal | 일상 기본 고양이 | Mochi, Byte |
| rare | 색 variant, 차분한 대안, 대부분의 시즌 | Sakura Cat, Rudolph Cat |
| unique | 강한 개성 액세서리/실루엣 | Snow Cat, Pumpkin Cat |
| legendary | 장기 사용·시즌 정점 | Aurora Cat |
| (mythic) | 추후 확장 자리 | — |

등급은 **순서 있는 enum**으로만 비교하고 가중치는 **테이블 1개**로 둔다. 위에 `mythic`을 더해도 스키마 변경이 없도록 설계한다.

### 18.2 두 단계 추첨

| 단계 | 조건 | 동작 |
|---|---|---|
| 수집 단계 | 미보유 기본 캐릭터 있음 | 미보유 중에서만 등급 가중 추첨. 중복 없음 → §11의 무중복 보장 유지 |
| 수집 후 단계 | 모든 기본 보유 | 풀 가중 추첨. 중복은 `알 조각`으로 환산 |

### 18.3 "어려운 알은 늦게" 보장

- **anti-frontload 게이트**: `unique`는 알 4개째부터, `legendary`는 알 8개째 또는 누적 500M tokens 이후 후보에 편입.
- **pity**: rare+ 가 5연속 없으면 바닥 가중치 상향, legendary는 30 알 천장에서 확정. 후반 운빨로 영영 못 받는 것을 막아 성장감을 지킨다.

### 18.4 중복 환산

수집 후 단계에서 보유 캐릭터가 뽑히면 `알 조각`으로 바꾼다. `알 조각`은 이미 방석/level up 연출 모티프로 존재하므로 새 화폐를 만들지 않고 재사용한다. 일정량을 모으면 스킨/액세서리 해금에 쓴다.

## 19. 최종 추천

기본 캐릭터는 `Mochi`로 확정하는 것이 좋다.

이유:

1. 고양이는 첫인상에서 사람을 끌어들이는 힘이 강하다.
2. 알에서 깨어나는 구조가 온보딩과 성장감을 만든다.
3. 메뉴바 작은 크기에서도 귀/얼굴 실루엣이 잘 읽힌다.
4. 잠, 집중, 타이핑, 지침 같은 상태 표현이 자연스럽다.
5. 스킨과 두 번째 고양이로 취향 확장이 쉽다.

캐릭터의 핵심은 "계속 옆에 둬도 질리지 않는 작은 존재감"이다. TokenMac의 기본 companion은 토큰 소비를 재촉하는 캐릭터가 아니라, 사용자의 작업 흔적을 먹고 조용히 자라는 메뉴바 고양이여야 한다.
